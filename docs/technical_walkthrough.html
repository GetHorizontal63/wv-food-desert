Technical Walkthrough

A detailed account of the data pipeline, retailer locator processing, road-network distance computation, food desert polygonization, and SNAP participation analysis used to map the collapse of food retail infrastructure in the southern West Virginia coal counties. All code is Python; all inputs are public-domain or open-licensed.

1. Data Sources & Acquisition

The pipeline uses seven public or open-licensed datasets. No proprietary feeds or paid API keys are required for core operation.

USDA-FNS Historical SNAP Retailer Locator (2005–2025) is the primary input for every map. The file records every SNAP-authorized retailer that has ever operated in the United States, with authorization date, end date, store category, geocoded address, and a Record ID. It is distributed as a zipped CSV and must be manually placed at data/raw/snap_retailers_2005_2025.zip before the pipeline runs. The pipeline never modifies this file; all transformations are computed at runtime.

TIGER/Line 2020 county boundaries are downloaded directly from the Census Bureau's public file server (tl_2020_us_county.zip) during pipeline.acquire. The file is filtered to WV FIPS 54 and then to the 19 named study counties.

OpenStreetMap via Overpass API supplies the road network (motorway through unclassified ways, cached at data/raw/osm_roads_overpass.json) and current grocery and dollar-store point locations for supplemental layers. Queries fan across three public mirror endpoints with automatic fallback.

Microsoft USBuildingFootprints (West Virginia) provides ML-extracted building polygons with substantially better rural coverage than OSM alone. The pipeline downloads the state file, filters to the study area bounding box, and clips to county polygons. Approximately 382,000 residential structures survive after filtering.

Mapzen/Nextzen Terrain Tiles via AWS Open Data supply elevation data for the hillshade backdrop. Tiles are fetched as 512×512 float32 GeoTIFFs in Web Mercator at zoom level 9 (~120 m resolution at WV latitude), cached individually to data/raw/terrain_tiles/. No authentication required.

U.S. Census Bureau ACS 5-year (2018–2022), Table B22003 provides SNAP household participation rates at the Census tract level, fetched via the Census API without a key. Tract polygons are drawn from TIGER/Line 2022, also fetched directly from Census servers.

MSHA Mine Safety and Health Administration open datasets supply coal employment figures for the coal-decline choropleth. Two pipe-delimited files are downloaded from MSHA's open government data portal: Mines.zip (mine master with county FIPS and commodity type) and MinesProdYearly.zip (annual average employee count and coal production by mine and subunit), joined on MINE_ID and filtered to WV coal commodity.

All outputs use EPSG:4326. Distance calculations use EPSG:32617 (UTM Zone 17N).

2. Pipeline Overview

The pipeline is a collection of independent Python modules under the pipeline/ package, each reading from and writing to data/processed/. Any module can be re-run independently as long as its upstream outputs exist.

python -m pipeline.acquire            # county boundaries; OSM grocery and dollar-store points
python -m pipeline.roads              # OSM road network GeoJSON for visual rendering
python -m pipeline.retail             # clip OSM grocery points to study area
python -m pipeline.dollar_store       # clip OSM dollar-store points to study area
python -m pipeline.grocers_by_year    # SNAP retailer snapshots 2005, 2010, 2015, 2020, 2025
python -m pipeline.buildings          # residential building footprints + access flags
python -m pipeline.isochrone          # drive graph build + isochrone generation
python -m pipeline.buildings_drive    # multi-source Dijkstra drive distances per home
python -m pipeline.food_deserts       # FNS desert polygon generation, 2005 and 2025
python -m pipeline.snap               # ACS SNAP participation by Census tract
python -m pipeline.coal               # MSHA coal employment time-series
python -m pipeline.population         # decennial population time-series
python -m pipeline.hillshade          # colored relief PNG from terrain tiles
python -m pipeline.grocery_history    # CBP grocery establishment counts (supplemental)
python -m pipeline.health             # HRSA HPSA shortage-area overlay (supplemental)
python -m pipeline.intervention       # greedy store-placement optimizer (supplemental)
python -m pipeline.export             # bundle all layers into web_bundle.json

The drive graph built by pipeline.isochrone is cached to data/raw/wv_south_drive.graphml; the initial build takes 2–5 minutes. Subsequent runs load from cache without re-querying Overpass.

3. Study Area & County Boundaries

The study area is all West Virginia counties at or south of the latitude of Kanawha County. The canonical list of 19 counties is defined in config.py: Kanawha, Boone, Lincoln, Logan, Mingo, McDowell, Wyoming, Mercer, Summers, Monroe, Greenbrier, Pocahontas, Nicholas, Webster, Braxton, Clay, Roane, Fayette, and Raleigh.

Fayette and Raleigh were added beyond the initial draft list as a judgment call — they are core southern coalfield counties (Beckley, Oak Hill, Fayetteville) whose exclusion would materially understate the food-desert footprint. The pipeline documentation explicitly flags this as a discretionary inclusion. TIGER county polygons are filtered to WV FIPS 54 and restricted to the named list. The resulting southern_wv_counties.geojson serves as the clip polygon for every downstream layer.

4. SNAP Retailer Locator Processing

Snapshot Construction

The USDA-FNS Historical SNAP Retailer Locator contains one row per authorization event. A store authorized in 2003 and closed in 2011 appears once, with Authorization Date = 2003-xx-xx and End Date = 2011-xx-xx. Five point-in-time snapshots are materialized by retaining records where the authorization date falls on or before January 1 of the snapshot year and either the end date is null or falls on or after that date:

snap_dt = pd.Timestamp(f"{year}-01-01")
active = df[
    (df["auth_date"].notna()) & (df["auth_date"] <= snap_dt) &
    ((df["end_date"].isna()) | (df["end_date"] >= snap_dt))
]
active = active.drop_duplicates(subset=["Record ID"], keep="first")

Coordinates outside the plausible WV bounding box (latitude 37–41°N, longitude 83–77°W) are dropped as geocoding failures before snapshot construction.

Store Category Hierarchy

The FNS file uses a free-text Store Type field. The pipeline retains six categories and excludes all others (convenience stores, pharmacies, farmers markets, wholesalers, specialty stores):

Retained typeRole in analysis
SupermarketFull-service tier; Dijkstra source for drive-distance and desert calculations
Super StoreFull-service tier; Dijkstra source for drive-distance and desert calculations
Large Grocery StoreCounted in full-service totals; not used as a Dijkstra source
Medium Grocery StoreCounted in full-service totals; not used as a Dijkstra source
Small Grocery StoreCounted in full-service totals; not used as a Dijkstra source
Combination Grocery/OtherCounted in headline totals only; excluded from distance and desert calculations

The headline total climbs from 604 to 987 between 2005 and 2025 solely because USDA expanded SNAP authorization to the Combination/Other tier after 2008. The full-service tier fell from 224 to 159 (−29%) over the same period. Large grocery stores alone fell from 32 to 12 (−63%).

Replacement Event Detection

The animated Step 2 map distinguishes replacement events from net closures. A closure at a given parcel is classified as a replacement if a different operator opens at the same location (coordinates rounded to three decimal places, ~110 m at WV latitude) within the same five-year window — the dollar-store-takes-the-supermarket-lot pattern. Replacement events are rendered in amber; pure closures in red; pure openings in green.

5. Residential Building Footprints

Microsoft USBuildingFootprints are distributed as a state-level GeoJSON. The WV file is filtered to the study area bounding box on download, then clipped to the county boundary polygon via GeoPandas spatial join. Building centroids are derived with .centroid in EPSG:32617 and reprojected to EPSG:4326 for the distance step.

Residential classification joins each building centroid to OpenStreetMap building use-tags. Buildings tagged building=yes (unclassified) are retained alongside explicit residential tags (house, residential, apartments, detached, semidetached_house). Commercial, industrial, and civic tags are dropped. Unmatched buildings — those not covered by any OSM polygon — are retained on the assumption that the overwhelming majority of rural structures in the southern coalfield are residential. Centroid longitude and latitude are stored as feature properties in addition to geometry so the export stage can emit compact parallel arrays without re-parsing coordinates.

6. Road Network Construction

The drive graph is built by pipeline.isochrone using osmnx. The query polygon is the county union buffered by 0.25 degrees (~25 km) to prevent edge effects at county lines. After download, osmnx.add_edge_speeds() and osmnx.add_edge_travel_times() enrich every edge with speed estimates from posted limits where tagged, or OSM highway-class defaults where not.

A separate lighter road GeoJSON for visual rendering is produced by pipeline.roads, which queries Overpass directly for way geometries (motorway through unclassified) and classifies them into four display tiers. The rendering layer and the routing graph are built from the same OSM source but maintained separately: the routing graph needs travel-time attributes and full node topology; the rendering layer needs only line geometries and a road class label. After clipping to the county union polygon, Shapely's intersection can produce stray point or geometry-collection artifacts at boundary tangencies; these are filtered to keep only LineString and MultiLineString types before export.

7. Drive-Distance Computation

Drive distances are computed along the actual road network rather than crow-flies. In southern WV terrain, a half-mile of ridge between a home and a closing store can mean fifteen miles of switchbacks — straight-line distance systematically understates the access problem.

Graph Simplification

pipeline.buildings_drive loads the cached MultiDiGraph and collapses it to a simple undirected nx.Graph weighted by edge length in meters. For any node pair connected by multiple directed edges, only the minimum-length edge is retained. Treating the network as bidirectional is valid for our purpose — computing distance to the nearest store, not modeling one-way traffic — and simple graphs run Dijkstra substantially faster than multigraphs.

Multi-Source Dijkstra

Every node with valid x/y attributes is indexed into a scipy.spatial.cKDTree over longitude/latitude coordinates. Active Supermarket and Super Store locations are snapped to graph nodes and de-duplicated. A single NetworkX multi-source Dijkstra traversal from all active store nodes simultaneously produces the shortest road-network distance in meters from every reachable node to its nearest store:

dist_by_node = nx.multi_source_dijkstra_path_length(
    H, sources=set(sm_nodes), weight="length"
)

Each home's drive distance is the Dijkstra distance at its snapped node plus a haversine residual for the straight-line walk from the building centroid to the snapped node. Buildings whose snapped node is unreachable — disconnected subgraph components, typically isolated private roads — receive a null distance and are excluded from delta calculations and summary statistics.

Isochrone Generation

pipeline.isochrone generates 15, 30, and 45-minute drive-time polygons using a three-tier fallback. Tier 1 attempts a locally-running Valhalla instance at http://localhost:8002. Tier 2 falls back to the OSMnx ego-graph approach on the cached drive graph: for each store node, nx.ego_graph extracts all nodes reachable within the travel-time radius, and the convex hull of those node coordinates (smoothed with a small buffer-out/buffer-in pass) becomes the isochrone polygon. Tier 3 is a Euclidean buffer at RURAL_AVG_KMH = 48 (~30 mph), used as a last resort and labeled buffer_fallback in output properties.

8. Per-Home Distance Delta

After Dijkstra runs for 2005 and 2025, the change in driving distance per home is delta = drive_mi_2025 − drive_mi_2005. Positive values indicate homes that got farther from a full-service grocer; negative values indicate homes that got closer. The delta array is written as a parallel sidecar at data/processed/buildings_drive_distances.json in the same feature order as buildings_residential.geojson, allowing the export stage to join by array index without a spatial merge.

The color scale uses a square-root transform on the absolute delta before mapping to the diverging ramp, spreading the common case of modest worsening (0–2 miles) across more of the visual range than a linear scale. The midpoint color is slate grey (#b8c2cc) rather than white to separate it visually from the white road network overlaid on the same map.

DELTA_CLAMP = 20.0
t = sqrt(abs(delta) / DELTA_CLAMP)     # [0, 1], sqrt-scaled
color = green_scale(t) if delta < 0 else red_scale(t)

Six summary statistics are computed from the delta array and stored in delta_stats for the frontend legend: n_worse_half_mi, median_lost_mi, n_better_half_mi, median_gained_mi, 10th/90th percentiles of the full distribution, and a distance_kind flag ("drive" when the Dijkstra sidecar is present, "crow_flies" on fallback).

9. Food Desert Polygonization

Threshold Definition

The FNS dual threshold classifies land as a food desert based on driving distance to the nearest Supermarket or Super Store. The thresholds were calibrated by the USDA Economic Research Service against the National Household Food Acquisition Survey:

  • Inside a Census Urban Area (Charleston, Beckley, Bluefield, Huntington fringe): food desert if drive distance exceeds ½ mile. Approximates the distance at which urban shoppers without cars substitute away from full-service stores.
  • Outside Census Urban Areas: food desert if drive distance exceeds 10 miles. The round-trip distance at which rural households begin substituting convenience stores for full grocery runs.

Grid Construction & Polygonization

Rather than testing building centroids (which would produce a point cloud), the desert layer is computed on a regular grid at approximately 0.35-mile (~560 m) spacing across the study area so the output can be polygonized into continuous area coverage. Each grid point inherits drive distance from the same multi-source Dijkstra outputs used in Section 7. The boolean mask is rasterized to a GeoTIFF at the grid resolution and polygonized with rasterio.features.shapes:

from rasterio.features import shapes
mask_shapes = list(shapes(desert_mask.astype("uint8"), transform=grid_transform))
desert_polys = [shape(s) for s, v in mask_shapes if v == 1]

Intersection and difference operations for change classification run on the rasterized masks before polygonization, not on the output polygons, to avoid topological inconsistencies from polygon overlay at this resolution.

Change Classification

Three polygon layers are produced by comparing the 2005 and 2025 masks. Desert disappeared (blue) covers cells that were desert in 2005 but not in 2025 — areas where a new store brought previously underserved land within threshold distance. Desert grew (red) covers cells that became desert between 2005 and 2025 — where store closures pushed previously served land beyond the threshold. Persistent desert (purple) covers cells that were desert in both years. The purple persistent category dominates by area, representing land structurally underserved for the full twenty-year study period.

Area statistics for the grown and disappeared categories are stored in food_deserts._meta.summary and consumed by the frontend to draw to-scale nested comparison squares in the Step 5 legend panel. Square side lengths are computed at runtime from current map pixels-per-mile so they remain proportionally accurate at any zoom level.

10. SNAP Participation Choropleth

SNAP participation rates are drawn from ACS 5-year 2018–2022, Table B22003 (Receipt of Food Stamps/SNAP in the Past 12 Months by Poverty Status), fetched from the Census API without a key. snap_pct = B22003_002E / B22003_001E × 100. Tracts with zero total households or null estimates are dropped. The 193-tract result is joined to TIGER 2022 tract polygons on GEOID and clipped to the study area. TIGER polygons are downloaded directly from Census servers as a zipped shapefile; no local copy is required before the pipeline runs.

The r = −0.56 Pearson correlation between county-level SNAP enrollment rate and supermarkets per 10,000 SNAP households is computed at the county level, not the tract level, to smooth over the high within-county variance in tract-level SNAP rates. It confirms the central claim of Step 6: the places where the most households need SNAP are the same places where the program's intended retail channel is thinnest.

11. Terrain & Hillshade

The colored relief backdrop is generated by pipeline.hillshade from Mapzen/Nextzen Terrain Tiles served from AWS Open Data at zoom level 9 (~120 m resolution). Tiles are mosaicked in their native Web Mercator projection with rasterio.merge, reprojected to EPSG:4326 with bilinear resampling, and composited using a custom six-stop hypsometric colormap (deep blue-green valleys → olive → tan → orange ridges → near-cream peaks).

Hillshade is computed at azimuth 315° (northwest sun) and altitude 45°. The compositing formula blends shade and color as rgb × (0.35 + 0.65 × shade), preventing shadows from going pitch black. The result is alpha-masked to the county union polygon via rasterio.features.geometry_mask and saved as an RGBA PNG alongside a bounds JSON file consumed by Leaflet's imageOverlay.

12. Visualization & Export

pipeline.export assembles all processed layers into a single web/data/web_bundle.json for the frontend. All polygon layers are simplified with a 0.0005-degree tolerance via shapely.simplify to reduce file size. The grocers-by-year layer uses a raw JSON pass-through rather than GeoPandas serialization to preserve string properties (authorization dates, end dates) that GeoPandas would coerce to Timestamps. A JS shim variant (web_bundle.js) assigns the payload to window.__WV_FOOD_DESERT_BUNDLE__ so the page renders when opened via file:// without a local server, since browsers block fetch() against file-protocol origins but permit <script src=...>.

The residential buildings layer (~382,000 points) is emitted as a separate compact columnar file (buildings_residential.js) — parallel arrays of longitudes, latitudes, and per-home distance deltas rather than GeoJSON features, approximately 3× smaller than an equivalent FeatureCollection. A summary object carrying aggregate statistics is embedded in the main bundle so the frontend legend renders correctly before the heavy file finishes loading.

The buildings layer is rendered on an HTML5 Canvas element rather than as SVG circle markers to avoid the DOM overhead that makes 382,000 circleMarker elements unusable. The Step 4 dim transition pre-sorts buildings at build time into a keep-set (homes that got at least half a mile farther, always full opacity) and a fade-set (all others, animated to invisible over 600 ms via requestAnimationFrame) so the worsened homes emerge from the noise of the full dataset rather than snapping into view.

Map state is driven entirely by an IntersectionObserver watching each .step element at a 55% intersection threshold. Each step declares a desired layer key set; on step entry the system diffs currently-visible layers against the desired set, fades departing layers 1→0 via the Web Animations API over 600 ms, then adds entering layers and fades them 0→1. Hillshade and road panes are never removed from the DOM; their visibility is controlled by CSS opacity transitions on the pane elements rather than Leaflet's setOpacity, which bypasses CSS transitions and would cause snapping instead of easing.
